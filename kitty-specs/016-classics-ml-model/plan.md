# Implementation Plan: Classics ML Model

**Branch**: `016-classics-ml-model` | **Date**: 2026-04-02 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `kitty-specs/016-classics-ml-model/spec.md`

## Summary

Build an independent ML model to predict cycling classics outcomes, replacing the rules-based scoring system (baseline rho ~0.31). The approach: a decoupled pipeline with classic-specific features (same-race history, classic type affinity, seasonal pipelines, specialist profiles), rigorous A/B benchmarking against the rules-based baseline, and production integration upon GO. Research findings in [research.md](research.md).

## Architecture Decisions

### AD-1: Decoupled Pipeline

Classic prediction lives in its own files (`features_classics.py`, `predict_classics.py`, etc.) completely independent from stage-race code. This means:

- Zero risk of breaking stage-race predictions
- Freedom to experiment with radically different features
- Can be developed and benchmarked without touching production code
- Integration is a thin delegation layer (3 lines in predict_sources.py)

### AD-2: Single Regression Model (not 4-source)

Unlike stage races which decompose into GC + stage + mountain + sprint sources, classics only have one scoring category (GC position). A single model predicts total expected points (0-200 scale). Simpler, fewer failure modes.

### AD-3: Feature Tiering with Ablation

Features are organized in 3 tiers (core → domain → experimental). Each tier and each experimental feature is A/B tested individually against the previous best, using the same benchmark protocol. This prevents "kitchen sink" overfitting.

### AD-4: Classic Type Taxonomy as Code

The lookup table mapping race_slug → types is a Python dict in `classic_taxonomy.py`, not a database table. Reasons: it's small (~20 entries), static (classic calendar barely changes year to year), and easier to version control.

### AD-5: Benchmark-First Development

Every code change goes through the benchmark before being accepted. The development loop is:

1. Write feature extraction code
2. Cache features
3. Run benchmark
4. Compare vs baseline (delta on all metrics)
5. Accept or reject

### WP01: Baseline Benchmark (P1)

**Goal**: Establish the true rules-based baseline for classics with comprehensive metrics.
**Scope**: Create `benchmark_classics.py` with rules-based scoring evaluation, reuse metric functions from benchmark_v8.py, output logbook entry with per-race breakdown.
**Files**: `ml/src/benchmark_classics.py` (new), imports from `ml/src/points.py` and `ml/src/benchmark_v8.py`

**Estimated complexity**: Medium

### WP02: Classic Taxonomy + Core Feature Extraction (P1)

**Goal**: Build the classic type lookup table and core feature extraction pipeline.
**Scope**: Create `classic_taxonomy.py` with type definitions and pipeline groupings. Create `features_classics.py` with Tier 1 features (same-race history, classic-specific points, rates, micro-form, team). Create `cache_features_classics.py` for caching.

### WP04: Domain Feature Engineering (P1)

## Risk Assessment

| Risk                                      | Likelihood | Impact | Mitigation                                                                                                          |
| ----------------------------------------- | ---------- | ------ | ------------------------------------------------------------------------------------------------------------------- |
| ML model doesn't beat baseline            | Medium     | Low    | The research is valuable regardless — we learn what works and what doesn't. Rules-based fallback remains.           |
| Overfitting to small dataset              | Medium     | Medium | Expanding-window CV, ablation testing, feature tiering prevent accepting spurious improvements.                     |
| Classic taxonomy incomplete               | Low        | Low    | Lookup table is easily updated. Unknown race_slugs fall back to generic features.                                   |
| Sparse same-race history for new riders   | Medium     | Low    | Core features include general racing form (pts_30d, team_rank) as fallback when classic-specific history is absent. |
| Integration breaks stage race predictions | Very Low   | High   | Fully decoupled pipeline. Only 3 lines changed in existing files.                                                   |
