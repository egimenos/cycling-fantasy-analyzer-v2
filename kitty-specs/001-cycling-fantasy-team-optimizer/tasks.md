## Work Package WP05: Scoring Engine (Priority: P1) 🎯 MVP

**Goal**: Implement the pure domain scoring engine — temporal decay weighting, per-category projected score computation, composite rider scoring, AND price-aware value scoring (compositeScore = f(projectedPts, price, pool)). 100% test coverage required.
**Independent Test**: Given a set of RaceResults for a rider and a rider pool with prices, the engine correctly computes projected scores with temporal decay AND composite scores that capture the price-quality relationship.
**Prompt**: `tasks/WP05-scoring-engine.md`
**Estimated Prompt Size**: ~400 lines

**Requirements Refs**: FR-004, FR-004b, FR-006

### Included Subtasks
- [x] T023 Implement scoring weights configuration (`domain/scoring/scoring-weights.config.ts`) — Grandes miniVueltas point values per position for each category (GC: 200pts for 1st..., mountain HC: 12pts..., sprint: 6pts..., daily stage: 15pts...)
- [x] T024 Implement temporal decay logic in scoring service — weight multipliers: current season ×1.0, previous ×0.6, two seasons ago ×0.3
- [x] T025 Implement per-category projected score computation (`domain/scoring/scoring.service.ts`) — for each category, compute weighted average of historical positions mapped to points
- [x] T026 Implement composite rider score aggregation — sum all category projections into `totalProjectedPts` (pure performance), AND implement `computeCompositeScore` that produces a price-aware value score relative to the rider pool (FR-004b: score = f(price, historical performance))
- [x] T027 Write 100% coverage unit tests — edge cases: rider with no data (0 score), single season only, all DNFs, rider only in classics (no GC data), mixed race types with filtering

### Implementation Notes
- Scoring service is a **pure function**: `(raceResults: RaceResult[], targetRaceType: RaceType) => RiderScore`
- NO I/O, NO side effects, NO framework imports — lives entirely in `domain/scoring/`
- Scoring weights are a static configuration object, not DB-driven
- For position → points mapping: 1st gets max points, positions beyond the point threshold get 0
- Handle DNF: position = null → contributes 0 points but counts toward the denominator for averaging

### Parallel Opportunities
- T023 and T024 can proceed in parallel (config vs. logic)

### Dependencies
- Depends on WP02 (needs domain entity types for RaceResult, RiderScore)

### Risks & Mitigations
- Scoring formula edge cases: riders with results in wrong race type should be filtered out, not crash
- Constitution mandates 100% coverage — write tests FIRST, then implement

## Work Package WP07: Team Optimizer (Priority: P2)

**Goal**: Implement the knapsack DP optimization algorithm for selecting the best 9-rider team within budget, with support for locked/excluded riders and alternative teams.
**Independent Test**: Given scored riders and budget, the optimizer returns valid 9-rider teams that respect budget and lock/exclude constraints, with the optimal team scoring higher than random selection.
**Prompt**: `tasks/WP07-team-optimizer.md`
**Estimated Prompt Size**: ~350 lines

**Requirements Refs**: FR-005, FR-007, FR-008, FR-010, FR-012

### Included Subtasks
- [x] T033 Implement knapsack DP algorithm (`domain/optimizer/knapsack.service.ts`) — 0/1 knapsack with dual constraint: team size = 9, total cost ≤ budget; returns optimal rider combination
- [x] T034 Implement must-include/must-exclude constraint handling — pre-filter rider pool, subtract locked rider costs from budget, validate feasibility before running DP
- [x] T035 Implement alternative team generation — find top 5 distinct team combinations by iteratively excluding riders from the optimal set and re-running
- [x] T036 Implement optimize team use case + POST /api/optimize endpoint with DTOs per contracts/api.md
- [x] T037 Write 100% coverage unit tests for knapsack — edge cases: budget too low, all riders locked exceed budget, fewer than 9 eligible riders, all same price, single rider per team slot

### Implementation Notes
- Knapsack is a pure domain function: `(riders: ScoredRider[], budget: number, teamSize: number, mustInclude: string[], mustExclude: string[]) => TeamSelection[]`
- DP table dimensions: riders × budget × team_size — O(200 × 2000 × 9) ≈ 3.6M cells — fast enough
- Alternative teams: after finding optimal, mask one rider at a time from optimal set, re-run DP, collect unique teams, sort by score, take top 5
- Response includes `scoreBreakdown` per team (gc, stage, mountain, sprint, daily)

### Parallel Opportunities
- T033 and T034 can be developed together (constraint handling feeds into DP initialization)

### Dependencies
- Depends on WP06 (needs AnalyzedRider type and scoring output)

### Risks & Mitigations
- DP memory usage: 3.6M cells × 8 bytes ≈ 29MB — well within limits
- Alternative team generation: naive approach (re-run DP 9 times) may produce duplicates — deduplicate by team composition hash
