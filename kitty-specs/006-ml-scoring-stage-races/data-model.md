# Data Model: ML Scoring for Stage Races

**Feature**: 006-ml-scoring-stage-races
**Date**: 2026-03-20

## New Entity: ML Score (Cache)

### Table: `ml_scores`

This table serves as a **prediction cache** — the ML service writes predictions on-demand when first requested, and the TypeScript API reads them for subsequent requests.

| Column          | Type      | Constraints                   | Description                                                  |
| --------------- | --------- | ----------------------------- | ------------------------------------------------------------ |
| id              | UUID      | PK, default gen_random_uuid() | Primary key                                                  |
| rider_id        | UUID      | FK → riders.id, NOT NULL      | The rider this prediction is for                             |
| race_slug       | VARCHAR   | NOT NULL                      | Race identifier (e.g., "tour-de-france")                     |
| year            | INTEGER   | NOT NULL                      | Race year                                                    |
| predicted_score | REAL      | NOT NULL                      | ML model predicted score (total projected points)            |
| model_version   | VARCHAR   | NOT NULL                      | Timestamp-based version identifier (e.g., "20260320T030000") |
| created_at      | TIMESTAMP | NOT NULL, default now()       | When this prediction was generated                           |

### Constraints

- **UNIQUE**: `(rider_id, race_slug, year, model_version)` — one prediction per rider per race per model version
- **INDEX**: `(race_slug, year, model_version)` — fast lookup for "all predictions for a race with current model"
- **INDEX**: `(model_version)` — fast filter by model version for cleanup/comparison

### Relationships

```
riders (existing)
  │
  └─── ml_scores (NEW, cache)
         rider_id FK → riders.id
```

No relationship to `race_results` or `startlist_entries` — the ML service joins those tables during feature extraction, but `ml_scores` only stores the cached prediction output.

### Cache Behavior

- **Write**: ML service writes after generating predictions for a race (on first request or cache invalidation)
- **Read**: TypeScript API reads cached predictions for a race + model version
- **Invalidation**: When `model_version` in cache differs from current model version, the entry is stale. The API triggers a re-prediction via the ML service.
- **Cleanup**: Old model version entries may accumulate. Periodic cleanup (optional) removes entries with non-current model versions.

## Modified Entity: AnalyzedRider (shared-types)

### New Fields

| Field            | Type                  | Description                                                 |
| ---------------- | --------------------- | ----------------------------------------------------------- |
| scoringMethod    | `"rules" \| "hybrid"` | Which scoring method produced this result                   |
| mlPredictedScore | `number \| null`      | ML model prediction (null for classics or when unavailable) |

### Unchanged Fields

All existing fields remain: `compositeScore`, `totalProjectedPts`, `categoryScores` (gc, stage, mountain, sprint), `pointsPerHillio`, `seasonsUsed`, `seasonBreakdown`, etc.

## Modified Entity: BenchmarkResult (domain)

### New Fields

| Field             | Type             | Description                                                               |
| ----------------- | ---------------- | ------------------------------------------------------------------------- |
| mlSpearmanRho     | `number \| null` | Spearman rho using ML predictions (null for classics)                     |
| hybridSpearmanRho | `number \| null` | Spearman rho using hybrid method (ML for stage races, rules for classics) |

### Unchanged Fields

Existing `spearmanRho` is renamed to `rulesSpearmanRho` for clarity. All other fields remain.

## Model Artifact (filesystem, not DB)

### Path: `ml/models/`

| File                      | Format           | Description                      |
| ------------------------- | ---------------- | -------------------------------- |
| `model_mini_tour.joblib`  | joblib (sklearn) | Trained RF model for mini tours  |
| `model_grand_tour.joblib` | joblib (sklearn) | Trained RF model for grand tours |
| `model_version.txt`       | plain text       | Current model version identifier |

Models are gitignored. Generated by `make retrain`. Loaded into memory by the ML service at startup.

## Data Flow

```
User uploads           TypeScript API           ML Service (FastAPI)        PostgreSQL
price list             ──────────────           ────────────────────        ──────────
     │
     ├─→ analyze race
     │   ├─→ compute rules-based scores (existing)
     │   │
     │   ├─→ stage race?
     │   │   ├─→ check ml_scores cache ────────────────────────────────→ SELECT WHERE
     │   │   │   (race_slug, year, model_version)                        race_slug, year,
     │   │   │                                                           model_version
     │   │   ├─→ cache miss?
     │   │   │   ├─→ POST /predict ──────→ 1. Load race data
     │   │   │   │   {race_slug, year}     2. Extract 36 features
     │   │   │   │                         3. model.predict()
     │   │   │   │                         4. Write cache ──────────→ INSERT ml_scores
     │   │   │   │   ←── predictions ←──── 5. Return predictions
     │   │   │   │
     │   │   ├─→ cache hit? → read cached predictions
     │   │   │
     │   │   └─→ enrich AnalyzedRider with mlPredictedScore
     │   │
     │   ├─→ classic? → rules-based only
     │   │
     │   └─→ return response with scoringMethod + mlPredictedScore
```

## ML Service Endpoints

### POST /predict

```
Request:  { race_slug: string, year: number }
Response: { predictions: [{ rider_id: string, predicted_score: number }], model_version: string }
```

### GET /health

```
Response: { status: "healthy" | "no_model", model_version: string | null, models_loaded: string[] }
```
