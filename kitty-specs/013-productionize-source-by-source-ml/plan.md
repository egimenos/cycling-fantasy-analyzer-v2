# Implementation Plan: Productionize Source-by-Source ML Pipeline

**Branch**: `013-productionize-source-by-source-ml` | **Date**: 2026-03-29 | **Spec**: [spec.md](spec.md)

## Summary

Replace the monolithic Random Forest model (predicts single `predicted_score`) with the source-by-source pipeline from feature 012. The ML service already has FastAPI + Docker sidecar infrastructure — we adapt the internal model, extend the API response with a 4-source breakdown, and propagate through NestJS adapter to frontend.

## Technical Context

**Language/Version**: Python 3.11 (ML service), TypeScript (NestJS API + React frontend)
**Primary Dependencies**: FastAPI, scikit-learn (Ridge, LogisticRegression), pandas, psycopg2, joblib
**Storage**: PostgreSQL 16 (race_results, rider_ratings, ml_scores, rider_prices)
**Testing**: pytest (ML), Jest (API), Vitest (frontend)
**Target Platform**: Docker sidecar (internal), VPS via Dokploy
**Performance Goals**: Predictions for 176-rider GT in <30s
**Constraints**: ML service internal only, DDD/Hexagonal in NestJS, English-only repo

## Key Design Decisions

### D1: Prediction response format

The `/predict` endpoint response changes from:

```json
{ "predictions": [{ "rider_id": "uuid", "predicted_score": 285.0 }] }
```

To:

```json
{
  "predictions": [
    {
      "rider_id": "uuid",
      "predicted_score": 285.0,
      "breakdown": { "gc": 165, "stage": 80, "mountain": 12, "sprint": 28 }
    }
  ]
}
```

Backward-compatible: `predicted_score` is still present (sum of breakdown).

### D2: Model artifacts

Each sub-model saved as separate joblib file + one `metadata.json` with:

- Feature lists per model
- Gate thresholds
- Heuristic weights (sprint_final contender score)
- Glicko configuration
- Supply estimation config

This makes hot-reload work: new model version = reload all artifacts.

### D3: Heuristic models (GC position, sprint_final)

GC position ranking and sprint_final contender score are **not ML models** — they're deterministic heuristics with configured weights. These weights are stored in `metadata.json`, not as joblib files. The predict logic applies them directly.

### D4: What gets trained vs what is config

| Component                 | Type             | Artifact                     |
| ------------------------- | ---------------- | ---------------------------- |
| GC gate                   | Trained (LogReg) | `gc_gate.joblib`             |
| GC position               | Heuristic        | weights in `metadata.json`   |
| Stage flat/hilly/mountain | Trained (Ridge)  | `stage_{type}.joblib`        |
| Stage ITT gate            | Trained (LogReg) | `stage_itt_gate.joblib`      |
| Stage ITT magnitude       | Trained (Ridge)  | `stage_itt_magnitude.joblib` |
| Mountain final gate       | Trained (LogReg) | `mtn_final_gate.joblib`      |
| Mountain pass capture     | Trained (Ridge)  | `mtn_pass_capture.joblib`    |
| Sprint final              | Heuristic        | weights in `metadata.json`   |
| Sprint inter capture      | Trained (Ridge)  | `spr_inter_capture.joblib`   |

### D5: Cache / DB storage for breakdown

Option A: Add 4 columns to `ml_scores` (gc_pts, stage_pts, mountain_pts, sprint_pts).
Option B: Store breakdown as JSONB column.

**Decision: Option A** — explicit columns are queryable and type-safe. Only 4 extra columns.

### D6: Rules-based vs ML routing

The `MlScoringPort` interface stays the same conceptually. The adapter checks `race_type`:

- `grand_tour` / `mini_tour` → call ML service
- `classic` → return null (NestJS falls back to rules-based ScoringService)

This routing already exists implicitly (ML service returns empty for classics). Making it explicit in the adapter improves clarity.

### D7: Retraining pipeline

`make retrain` currently runs `retrain.py` which calls: load data → extract features → train RF → save.

New flow: load data → compute Glicko → build cache → build stage targets → build stage features → build classification features → train all sub-models → save artifacts → write version.

Single entry point: `python -m src.retrain` does everything.

```sql
ALTER TABLE ml_scores ADD COLUMN mountain_pts REAL DEFAULT 0;
ALTER TABLE ml_scores ADD COLUMN sprint_pts REAL DEFAULT 0;
```

### MlPrediction interface (extend)

```typescript
export interface MlPrediction {
  readonly riderId: string;
  readonly predictedScore: number;
  readonly breakdown: {
    readonly gc: number;
    readonly stage: number;
    readonly mountain: number;
    readonly sprint: number;
  };
}
```
