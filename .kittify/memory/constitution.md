- Rate-limited: 1.5s between PCS requests

**Scoring Engine**:

- Rules-based scoring: implemented within NestJS domain layer (`ScoringService`)
- ML scoring: Python FastAPI microservice (`ml/`) with scikit-learn Random Forest
- Quality over speed — correctness and accuracy take priority over response time

**ML Service** (`ml/`):

- Python 3.11 + FastAPI
- scikit-learn (Random Forest), pandas, psycopg2
- Internal Docker sidecar — not exposed externally
- Hot-reload on model version change (no container restart needed)
- Prediction cache in PostgreSQL (`ml_scores` table)
- Weekly retraining via `make retrain` (~5-10 min)

**Database**:

- PostgreSQL 16 (Alpine)
- Drizzle ORM (schema, migrations, queries)
- Shared types in `packages/shared-types`

## TypeScript Conventions

### Tooling Setup

All packages must include:

- **TypeScript** in strict mode (`"strict": true`)
- **ESLint** with TypeScript plugin
- **Prettier** for formatting
- **Husky** + **lint-staged** for pre-commit hooks

### General Best Practices

- **Immutability by default**: Use `const`, `readonly`, and `as const` whenever possible. Avoid direct mutation; prefer spread, `.map()`, `.filter()`
- **Avoid negation in boolean names**: Use `isActive` not `isNotDisabled`
- **Discriminated unions to model state**:
  ```typescript
  type AsyncState<T> =
    | { status: 'idle' }
    | { status: 'loading' }
    | { status: 'success'; data: T }
    | { status: 'error'; error: string };
  ```

### Naming Conventions

| Construct                  | Convention         | Example                               |
| -------------------------- | ------------------ | ------------------------------------- |
| Variables & functions      | `camelCase`        | `getUserProfile`, `isActive`          |
| Types, interfaces, classes | `PascalCase`       | `UserProfile`, `OrderService`         |
| Global/immutable constants | `UPPER_SNAKE_CASE` | `MAX_RETRY_ATTEMPTS`                  |
| Enum names & members       | `PascalCase`       | `OrderStatus.Confirmed`               |
| Files & folders            | `kebab-case`       | `user-profile.ts`, `order-service.ts` |

- **Do not prefix interfaces with `I`** — use `UserProfile` not `IUserProfile`

### Type System

- **Prefer `interface` for objects** (extensible, better compiler performance); **`type` for unions and compositions**
- **Avoid `any`** — use `unknown` with type guards instead
- **Always use `strict` mode** — no implicit `any`, no loose null checks
- **Explicit return types** on all exported/public functions
- **Leverage utility types**: `Partial<T>`, `Required<T>`, `Pick<T,K>`, `Omit<T,K>`, `Readonly<T>`, `Record<K,V>`, `ReturnType<F>`, `Parameters<F>`

```typescript
// Correct
interface User {
  id: string;
  email: string;
}
type Result<T> = { success: true; data: T } | { success: false; error: string };

export async function fetchUsers(): Promise<User[]> { ... }

// Incorrect
interface IUser { ... }           // No "I" prefix
function parse(input: any): any   // No any
```

## Tribal Knowledge

### Team Conventions

- **English only**: Every artifact in the repository (code, comments, docs, commits, ADRs) must be in English
- **No half-finished features**: Never merge incomplete work to `main`. Use feature branches. If a feature needs to be shipped partially, it must be independently functional and tested
- **Conventional Commits**: All commits must follow the Conventional Commits specification
- **DDD boundaries**: Domain logic never depends on NestJS decorators or infrastructure. If you need to test a domain rule, you should be able to do so with zero framework setup
- **Scoring model changes**: Any change to the scoring algorithm requires a 100% test coverage check and an ADR documenting the rationale

### Lessons Learned

- **ML target must match game scoring**: The ML model was trained on partial scoring data (~65% of game categories). Missing daily GC, mountain passes, and intermediate sprints distorted the training target. Always verify that training targets reflect the real-world metric being optimized.
- **PCS hidden tabs contain rich data**: Stage pages have 6 `div.resTab` elements — only Tab 0 (stage results) was being parsed. The hidden tabs contain daily classifications, individual pass results, and sprint data at no extra HTTP cost.
- **lint-staged + CRLF warnings**: lint-staged v16 fails with "Needed a single revision" when `git stash create` emits CRLF warnings to stdout. Fix: normalize line endings with `sed -i 's/\r$//'` before committing.
